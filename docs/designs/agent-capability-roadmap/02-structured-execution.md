# Agent 能力路线图 — 阶段一：结构化执行框架

> 原文拆分自 `../agent-capability-roadmap.md`。

## 4. 阶段一：结构化执行框架（P0）

> **目标**: 让 agent 具备「规划 → 执行 → 验证 → 反思」的基本能力
> **预期效果**: 能够可靠地完成 5 步以上的复杂任务
> **估算**: 2-3 天

### 4.1 设计要点

#### 4.1.1 规划模式（Planning Mode）

**触发条件**：
- 用户输入涉及「多步操作」（由系统提示词引导 Agent 自行判断）
- 或者显式请求「先规划再执行」

**实现方式**：
- 不需要修改 main.rs 循环结构（保持扁平）
- 而是通过**系统提示词 + 状态文件**来引导模型行为

**系统提示词新增指令**：

```
【结构化工作流程】
当你接收到需要多步执行的复杂任务时，请遵循以下流程：

1. 【规划阶段】先输出分析，然后创建 PLAN.md 文件：
   - 目标描述
   - 执行步骤（编号列表）
   - 每个步骤的验证标准

2. 【执行阶段】按 PLAN.md 的步骤逐个执行：
   - 每完成一步，更新 AGENDA.md 标记完成
   - 遇到错误时，先分析原因再修复
   - 重要发现记录到 MEMORY.md

3. 【验证阶段】每个代码修改后执行验证：
   - 修改 Rust 代码后 → `cargo check`
   - 修改配置文件后 → 检查语法完整性

4. 【总结阶段】所有步骤完成后向用户总结
```

#### 4.1.3 验证钩子（Verification Hook）

这是一个纯靠系统提示词引导的行为模式：

```
【验证规则】
每次修改 Rust 文件后，必须运行以下命令验证：
  cargo check 2>&1 | tail -20

如果编译失败：
  1. 分析错误信息
  2. 修复错误
  3. 再次验证
  4. 如果连续 3 次失败，回退到上一个已知可用状态并重新规划
```

### 4.2 需要修改的代码

#### 4.2.1 核心改动：系统提示词增强（最优先，零代码改动）

**文件**: `src/main.rs` 中 `ContextManager::new()` 的 system_prompt

只需要增强系统提示词，增加上述的结构化工作流程指令。**零代码改动，纯 Prompt Engineering**，效果立竿见影。

#### 4.2.2 可选改动：PLAN.md 状态追踪工具（次要）

新增一个 `plan` 工具（或增强 EditTool），让 agent 能更方便地更新状态文件：

| 工具 | 功能 |
|------|------|
| `plan` | 查看/创建/更新 PLAN.md |
| `checkpoint` | 记录当前状态快照 |
| `rollback` | 回滚到上一个 checkpoint |

但初期用 EditTool 操作 PLAN.md 完全够用。

### 4.3 验证标准

| 验证场景 | 预期行为 |
|---------|---------|
| "重构这个函数" | 先输出分析 + 计划，再执行 |
| 编译错误 | 自动分析错误并修复 |
| 中间步骤失败 | 尝试修复，失败后重新规划 |
| 5 步以上任务 | 全程跟踪状态，不断片 |
| 上下文压缩后 | 能通过 PLAN.md 恢复上下文 |
