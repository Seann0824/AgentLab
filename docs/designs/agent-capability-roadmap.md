## 1. 现状诊断

### 1.1 当前架构（简化）

```
┌─────────────────────────────────────────────────────────┐
│                     main.rs 主循环                        │
│                                                         │
│  User Input → LLM → Tool Calls → Execute → Results → LLM │
│                  ↑_________________________↓              │
│                    纯反应式循环 (Reactive Loop)            │
└─────────────────────────────────────────────────────────┘
```

这是一个 **Flat Loop**（扁平循环），没有任何层次结构：

| 方面 | 现状 | 问题 |
|------|------|------|
| **推理** | 模型每次只看到当前一步 | 没有全局规划，复杂任务容易迷失 |
| **状态** | 依赖上下文窗口保留消息 | 压缩后遗忘进度，没有持久化任务状态 |
| **自我修正** | 依赖模型自行判断 | 没有结构化「分析失败→调整」环节 |
| **工具调用** | 模型自行决定调什么 | 没有工具选择的策略层 |
| **验证** | 无 | 执行完不检查结果是否正确 |
| **自我进化** | 系统提示提到可以改代码 | 没有实际的工作流支持 |

### 1.2 已具备的基础设施

项目已有非常好的基础：

- ✅ **上下文窗口管理** (`ContextManager`) — 四层渐进压缩、preserve 机制、异步摘要
- ✅ **工具执行框架** (`ToolManager` + `Tool` trait) — 可扩展的工具注册
- ✅ **增量编辑工具** (`EditTool`) — search/replace/insert/delete/append
- ✅ **Shell 工具** (`BashShell`) — 命令执行
- ✅ **Token 估算** (`TokenEstimator`) — 增量缓存
- ✅ **模型适配器** (`ModelAdapter` trait) — 可插拔

核心缺失不在基础设施，而在**执行架构**。

## 2. 核心缺失分析

### 2.1 缺失一：没有规划层（Planning Layer）

**当前行为**：
```
用户: "帮我重构这个模块，使用依赖注入"
模型: (直接开始调 shell 读文件、调 edit 改代码...)
```

问题是：
- 模型没有先输出一个**全局计划**
- 改到一半上下文压缩后，模型忘了整体目标
- 无法判断"还需要几步完成"

**理想行为**：
```
用户: "帮我重构这个模块，使用依赖注入"
模型: 
  🧠 【分析】当前模块使用硬编码，需要改为 trait-based 依赖注入
  📋 【计划】
    1. 提取接口 trait
    2. 修改实现类实现 trait
    3. 在 main.rs 中注入依赖
    4. 验证编译通过
  🔧 【执行】
    开始第 1 步...
```

### 2.2 缺失二：没有验证循环（Verification Loop）

**当前行为**：
```
模型调 edit 改完代码 → 直接继续下一步
```

问题：
- 改坏了代码不知道
- 编译错误等到最后才发现
- 无法回滚

**理想行为**：
```
模型调 edit 改完代码
  → ✅ 自动执行 cargo check / cargo test
  → ❌ 编译失败！分析错误信息
  → 🔄 修复代码
  → ✅ 验证通过
  → 继续下一步
```

### 2.3 缺失三：没有状态持久化（State Persistence）

**当前行为**：
- 所有状态在 ContextManager 的消息列表里
- 压缩后早期消息被摘要替换
- 模型只能基于"当前上下文"工作

**理想行为**：
```
📄 AGENDA.md  — 当前任务的目标、步骤、完成状态
📄 PLAN.md    — 当前执行计划
📄 MEMORY.md  — 重要发现、决策记录
📄 LOG.md     — 执行日志
```

### 2.4 缺失四：没有工具选择策略（Tool Selection Strategy）

**当前行为**：
- 所有工具平等地暴露给模型
- 模型自行决定什么时候用什么工具

问题：
- 模型可能在需要 `edit` 时用了 `shell + sed`
- 没有"工具使用规范"约束

## 3. 总体架构设计

### 3.1 目标架构：分层 Agent 循环

```
┌────────────────────────────────────────────────────────────┐
│                     🧠 规划层 (Planner)                      │
│  分析需求 → 拆解步骤 → 输出计划 → 写入 PLAN.md               │
│  当计划完成时 → 生成总结报告                                  │
└────────────────────────────────────────────────────────────┘
          │ 计划 / 状态 ↑
          ▼              │ 反馈
┌────────────────────────────────────────────────────────────┐
│                     🔧 执行层 (Executor)                     │
│  按计划步骤执行 → 调工具 → 收集结果 → 验证                  │
│  每一步执行后更新 AGENDA.md                                 │
│  遇到错误 → 标记失败 → 请求重新规划                          │
└────────────────────────────────────────────────────────────┘
          │ 工具调用 ↑
          ▼              │ 结果
┌────────────────────────────────────────────────────────────┐
│                     🛠️ 工具层 (Tools)                       │
│  Shell  │  Edit  │  Read  │  Search  │  ... (可扩展)       │
└────────────────────────────────────────────────────────────┘
          │
          ▼
┌────────────────────────────────────────────────────────────┐
│                     📄 状态持久化 (Persistence)               │
│  AGENDA.md │ PLAN.md │ MEMORY.md │ LOG.md                  │
└────────────────────────────────────────────────────────────┘
```

### 3.2 核心循环设计

```
loop {
    // === 阶段 1: 接收输入 ===
    if 用户输入 or 计划完成:
        进入规划模式

    // === 阶段 2: 规划 ===
    if 规划模式:
        读取当前状态 (AGENDA.md, PLAN.md)
        LLM 输出分析 + 计划
        写入 PLAN.md
        写入 AGENDA.md (步骤清单)

    // === 阶段 3: 执行 ===
    if 有未完成步骤:
        读取当前步骤
        LLM 执行（带工具调用）
        验证结果（cargo check / 测试）
        更新 AGENDA.md (标记完成/失败)
        如果失败 → 请求重新规划/修复

    // === 阶段 4: 反思 ===
    if 所有步骤完成 or 遇到不可恢复错误:
        LLM 总结执行结果
        写入 MEMORY.md (经验沉淀)
        向用户汇报
}
```

## 4. 阶段一：结构化执行框架（P0）

> **目标**: 让 agent 具备「规划 → 执行 → 验证 → 反思」的基本能力
> **预期效果**: 能够可靠地完成 5 步以上的复杂任务
> **估算**: 2-3 天

### 4.1 设计要点

#### 4.1.1 规划模式（Planning Mode）

**触发条件**：
- 用户输入涉及「多步操作」（由系统提示词引导 Agent 自行判断）
- 或者显式请求「先规划再执行」

**实现方式**：
- 不需要修改 main.rs 循环结构（保持扁平）
- 而是通过**系统提示词 + 状态文件**来引导模型行为

**系统提示词新增指令**：

```
【结构化工作流程】
当你接收到需要多步执行的复杂任务时，请遵循以下流程：

1. 【规划阶段】先输出分析，然后创建 PLAN.md 文件：
   - 目标描述
   - 执行步骤（编号列表）
   - 每个步骤的验证标准

2. 【执行阶段】按 PLAN.md 的步骤逐个执行：
   - 每完成一步，更新 AGENDA.md 标记完成
   - 遇到错误时，先分析原因再修复
   - 重要发现记录到 MEMORY.md

3. 【验证阶段】每个代码修改后执行验证：
   - 修改 Rust 代码后 → `cargo check`
   - 修改配置文件后 → 检查语法完整性

4. 【总结阶段】所有步骤完成后向用户总结
```

#### 4.1.2 任务状态文件

##### PLAN.md — 执行计划

```markdown
# 执行计划: 模块重构-依赖注入

## 目标
将 `DatabaseService` 模块从硬编码重构为基于 trait 的依赖注入模式。

## 步骤

- [ ] 1. 分析当前代码结构
      - 验证: 生成当前模块的关系图
- [x] 2. 定义 `DatabaseTrait`
      - 文件: `src/db/trait.rs`
      - 验证: cargo check 通过
- [ ] 3. 实现 `PostgresDatabase`
      - 文件: `src/db/postgres.rs`
      - 验证: cargo check 通过
- [ ] 4. 修改 main.rs 使用依赖注入
      - 验证: cargo build 通过

## 状态
- 开始时间: 2025-01-15 10:00
- 当前步骤: 2
- 阻塞项: 无
```

##### AGENDA.md — 当前议程（精简版）

```markdown
# 当前议程

任务: 模块重构-依赖注入
进度: 2/4 步完成
当前步骤: 定义 `DatabaseTrait`
阻塞: 无
```

##### MEMORY.md — 经验与发现

```markdown
# Agent 记忆

## 项目结构
- src/db/ — 数据库相关模块
- src/main.rs — 入口文件

## 关键决策
- 【决策】2025-01-15: 选择 trait-based DI 而非泛型参数
  - 原因: 更好的测试性和可替换性

## 已知问题
- 【TODO】db 模块缺少错误处理单元
```

#### 4.1.3 验证钩子（Verification Hook）

这是一个纯靠系统提示词引导的行为模式：

```
【验证规则】
每次修改 Rust 文件后，必须运行以下命令验证：
  cargo check 2>&1 | tail -20

如果编译失败：
  1. 分析错误信息
  2. 修复错误
  3. 再次验证
  4. 如果连续 3 次失败，回退到上一个已知可用状态并重新规划
```

### 4.2 需要修改的代码

#### 4.2.1 核心改动：系统提示词增强（最优先，零代码改动）

**文件**: `src/main.rs` 中 `ContextManager::new()` 的 system_prompt

只需要增强系统提示词，增加上述的结构化工作流程指令。**零代码改动，纯 Prompt Engineering**，效果立竿见影。

#### 4.2.2 可选改动：PLAN.md 状态追踪工具（次要）

新增一个 `plan` 工具（或增强 EditTool），让 agent 能更方便地更新状态文件：

| 工具 | 功能 |
|------|------|
| `plan` | 查看/创建/更新 PLAN.md |
| `checkpoint` | 记录当前状态快照 |
| `rollback` | 回滚到上一个 checkpoint |

但初期用 EditTool 操作 PLAN.md 完全够用。

### 4.3 验证标准

| 验证场景 | 预期行为 |
|---------|---------|
| "重构这个函数" | 先输出分析 + 计划，再执行 |
| 编译错误 | 自动分析错误并修复 |
| 中间步骤失败 | 尝试修复，失败后重新规划 |
| 5 步以上任务 | 全程跟踪状态，不断片 |
| 上下文压缩后 | 能通过 PLAN.md 恢复上下文 |

## 5. 阶段二：自我进化能力（P1）

> **目标**: Agent 能够自主发现能力不足并扩展自身
> **预期效果**: Agent 可以「自举」——自己添加新工具、修复自己的 bug
> **估算**: 2-3 天

### 5.1 核心机制：内省（Introspection）

Agent 需要知道：
1. 我有哪些工具？ → 查看 `src/tools/` 目录
2. 工具是如何注册的？ → 查看 `initial_tool_manager()`
3. 如何添加新工具？ → 实现 `Tool` trait，注册到 `ToolManager`

#### 5.1.1 工具模板

提供一个**工具模板文件**，当 Agent 决定添加新工具时，可以复制模板并修改：

```rust
// src/tools/my_tool/mod.rs
// 模板：复制此文件并修改

use tokio::{sync::mpsc, fs};
use tokio_stream::wrappers::ReceiverStream;

use crate::tools::types::{Tool, ToolEvent, ToolStream};

pub struct MyTool;

impl Tool for MyTool {
    fn name(&self) -> &str { "my_tool" }
    fn description(&self) -> &str { "我的工具描述" }

    fn parameters_schema(&self) -> serde_json::Value {
        serde_json::json!({
            "type": "function",
            "function": {
                "name": "my_tool",
                "description": "...",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "input": { "type": "string" },
                    },
                    "required": ["input"],
                    "additionalProperties": false
                }
            }
        })
    }

    fn execute(&self, args: serde_json::Value) -> ToolStream {
        let (tx, rx) = mpsc::channel(1);
        tokio::spawn(async move {
            // 实现工具逻辑
            let _ = tx.send(ToolEvent::Done(serde_json::json!({
                "result": "done"
            }))).await;
        });
        Box::pin(ReceiverStream::new(rx))
    }
}
```

#### 5.1.2 注册步骤

Agent 添加新工具的步骤：
1. 创建 `src/tools/my_tool/mod.rs`
2. 在 `src/tools/mod.rs` 中注册：`pub mod my_tool;`
3. 在 `main.rs` 的 `initial_tool_manager()` 中注册：`tool_manager.register_tool(Box::new(MyTool));`
4. 运行 `cargo check` 验证

### 5.2 具体设计

#### 5.2.1 系统提示词中的自我进化指令

```
【自我进化】
你可以修改自身代码来增强能力：
1. 添加新工具 → 实现 Tool trait，注册到 ToolManager
2. 修改现有工具 → 增强已有工具的能力
3. 修复 bug → 修改对应 src/ 下的代码

规则：
- 修改代码后必须运行 cargo check 验证
- 验证通过后，新能力立即生效（下次调用时）
- 如果修改导致编译失败，分析错误并修复
- 重大改动建议分步进行（先加框架，再填实现）
```

#### 5.2.2 建议的初始工具集扩展

| 工具 | 优先级 | 说明 |
|------|--------|------|
| `read` (文件读取) | P1 | 更高效的文件读取，支持行号范围 |
| `search` (全文搜索) | P1 | 搜索目录中的文本内容 |
| `grep` (正则搜索) | P1 | 正则匹配搜索 |
| `plan` (计划管理) | P1 | 查看/更新 PLAN.md 状态 |
| `fetch` (网络请求) | P2 | HTTP GET 请求获取外部信息 |

## 6. 阶段三：多 Agent 协作（P2）

> **目标**: 支持多个 Agent 角色协同工作
> **预期效果**: 能处理需要不同专业知识的复杂任务
> **估算**: 3-5 天

### 6.1 架构设计

```
┌──────────────────────────────────────────┐
│           Orchestrator Agent              │
│  任务分解、分配、进度管理、结果汇总        │
└────┬────────┬────────┬────────┬───────────┘
     │        │        │        │
     ▼        ▼        ▼        ▼
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ Planner│ │ Coder │ │Tester │ │Reviewer│
│ 架构师 │ │ 编码  │ │ 测试  │ │ 审查   │
└──────┘ └──────┘ └──────┘ └──────┘
```

### 6.2 角色定义

| 角色 | 职责 | 工具集 |
|------|------|--------|
| **Orchestrator** | 任务分解、分配、协调 | 全部 |
| **Planner** | 需求分析、架构设计、计划输出 | read, search, plan |
| **Coder** | 编码实现、bug 修复 | read, edit, shell |
| **Tester** | 编写测试、运行测试、报告覆盖率 | shell, read |
| **Reviewer** | Code Review、质量检查 | read, search, shell |

### 6.3 实现方式

多 Agent 在初期可以通过「同一个模型实例，不同系统提示词」实现：

```
Agent 角色切换流程：
1. Orchestrator 分析任务
2. Orchestrator 写出子任务说明（写入文件）
3. 切换到 Coder Agent（加载 Coder 系统提示词）
4. Coder 执行完毕后，Orchestrator 验证结果
5. 切换到 Reviewer Agent...
```

更成熟的方案是使用多个 `ModelAdapter` 实例并行执行。

## 7. 阶段四：持久化与记忆（P3）

> **目标**: Agent 拥有跨会话的长期记忆
> **预期效果**: 多次对话之间能记住项目上下文、用户偏好
> **估算**: 3-5 天

### 7.1 记忆层级

```
┌──────────────────────────────────────────┐
│          工作记忆 (Working Memory)          │
│    当前任务状态、临时变量、步骤进度          │
│    载体: PLAN.md, AGENDA.md                │
├──────────────────────────────────────────┤
│          情景记忆 (Episodic Memory)         │
│    过去任务的经验、成功模式、错误教训        │
│    载体: MEMORY.md, 项目根目录的 .agent/   │
├──────────────────────────────────────────┤
│          语义记忆 (Semantic Memory)         │
│    项目结构知识、技术栈、约定规范            │
│    载体: .agent/knowledge.md               │
└──────────────────────────────────────────┘
```

### 7.2 存储结构

```
项目根目录/
├── .agent/                  # Agent 记忆目录
│   ├── memory.md            # 长期记忆（经验、决策、教训）
│   ├── knowledge.md         # 项目知识（结构、约定、技术选型）
│   └── sessions/            # 会话日志（可选）
│       └── 2025-01-15.md
├── PLAN.md                  # 当前执行计划
└── AGENDA.md                # 当前议程
```

### 7.3 关键设计：记忆检索

当 ContextManager 触发压缩时，将摘要内容同时写入 `.agent/memory.md`：

```
【注入时机】
ContextManager 压缩早期对话时
→ 同时将摘要追加到 .agent/memory.md
→ 下次会话启动时加载 memory.md 作为系统提示词的一部分
```

| 优先级 | 阶段 | 内容 | 预估工作量 | 效果 |
|--------|------|------|-----------|------|

| 🟢 **P2** | 三 | 多 Agent 协作 | 3-5天 | 专业分工处理复杂任务 |
| 🔵 **P3** | 四 | 持久化与记忆 | 3-5天 | 跨会话连续工作 |
